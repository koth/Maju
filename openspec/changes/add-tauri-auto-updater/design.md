## Context

Kodex is a Tauri v2 desktop app with the React UI in `apps/desktop/ui` and the shell in `apps/desktop/src-tauri`. The current release path is local packaging through `npm --prefix apps/desktop/ui run desktop:build`; there is no GitHub Actions release workflow and no updater plugin configured.

Tauri v2 updater requires signed artifacts. The public key is embedded in the app configuration and the private key is supplied to build jobs through `TAURI_SIGNING_PRIVATE_KEY`. GitHub Releases can host the static `latest.json` manifest generated by `tauri-action`, which fits the desired GitHub-like hosting model.

## Goals / Non-Goals

**Goals:**

- Enable Tauri v2 updater in the desktop app for Windows and macOS builds.
- Host update metadata and artifacts on the `koth/Kodex` GitHub Releases page.
- Add a user-triggered update flow that checks for updates, downloads, installs, and relaunches the app after confirmation.
- Add a GitHub Actions release workflow for Windows x64, macOS Intel, and macOS Apple Silicon.
- Document signing key generation, GitHub secrets, and tag-based release operation.

**Non-Goals:**

- Do not implement Linux updater/release jobs in this change.
- Do not implement a custom update server; use GitHub Release `latest.json`.
- Do not add Apple Developer ID notarization or Windows Authenticode signing in the first updater pass.
- Do not silently install updates while an ACP session may be running.

## Decisions

### Decision: Use Tauri updater with GitHub Release static JSON

Point the updater endpoint to `https://github.com/koth/Kodex/releases/latest/download/latest.json` in the base Tauri config, and enable `createUpdaterArtifacts: true` only through the release config used by CI.

Rationale: This uses Tauri's native signing and installer integration while keeping hosting operationally simple. `tauri-action` can generate the static manifest and upload signatures to the release, while local non-release builds do not require the updater private key.

Alternative considered: Build a custom update service. Rejected because there is no need for rollout rules or channel targeting yet.

### Decision: Keep update installation user-triggered

Expose update checking in the settings UI and let the user explicitly start installation and relaunch.

Rationale: Kodex can have running sessions, terminals, and unsaved editor buffers. A user-triggered flow avoids interrupting active work. A later change can add startup checks that only notify.

Alternative considered: Check and install automatically on launch. Rejected because Windows installers exit the application during installation and the editor has active user state.

### Decision: Use frontend updater APIs

Use `@tauri-apps/plugin-updater` from the React UI for `check()` and `downloadAndInstall()`, then `@tauri-apps/plugin-process` for `relaunch()`.

Rationale: The UI can naturally display update state and progress without introducing additional Rust commands. The Tauri plugin capability model already gates access.

Alternative considered: Add Rust commands wrapping updater state. Rejected for v1 because no cross-window or background scheduling is required.

### Decision: CI publishes one release from a matrix

Add `.github/workflows/release.yml` triggered by `app-v*` tags and manual dispatch. Build Windows x64, macOS x64, and macOS arm64 jobs, all uploading to the same GitHub Release through `tauri-apps/tauri-action`.

Rationale: GitHub-hosted runner labels distinguish macOS Intel and arm64 runners. Separate target builds keep release assets explicit and allow `latest.json` to include both `darwin-x86_64` and `darwin-aarch64`.

Alternative considered: Build a universal macOS app only. Rejected for the first pass because separate architecture outputs are easier to reason about with updater platform keys.

### Decision: Prefer NSIS for Windows updater metadata

Set `updaterJsonPreferNsis: true` in `tauri-action`.

Rationale: The project already has an NSIS postinstall hook for bundled agent binaries. The updater should exercise the same installer path as the primary Windows setup executable.

## Risks / Trade-offs

- A placeholder updater public key will not validate real updates. → Document the key generation step and require replacing `KODEX_UPDATER_PUBLIC_KEY_PLACEHOLDER` before publishing an updater-enabled release.
- Unsigned/notarization-free macOS artifacts may still trigger Gatekeeper warnings. → Keep this out of scope but document that updater signing is separate from platform code signing.
- GitHub Release `latest.json` always points to the latest published release. → Use draft releases for validation before publish, and tag app releases with `app-v<semver>`.
- CI builds bundled agent binaries for each platform and can be slow. → Use npm and Rust caches in the workflow.

## Migration Plan

1. Generate an updater key pair locally with `npx tauri signer generate -w ~/.tauri/kodex.key`.
2. Commit the generated public key into `tauri.conf.json`; store the private key content in GitHub secret `TAURI_SIGNING_PRIVATE_KEY`.
3. Release one updater-enabled version manually for each supported platform.
4. Future `app-v*` tags publish signed updater artifacts through GitHub Actions.
